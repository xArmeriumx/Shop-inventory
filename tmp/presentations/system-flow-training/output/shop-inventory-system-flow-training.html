ภาพรวม Architecture

Next.js monolith ที่แยก UI, action, service, DB และ core control ชัดเจน

Key takeawayService layer คือสมองของระบบ ส่วน Server Actions คือประตูรับข้อมูลจาก UIMermaid: scratch/mermaid/architecture.mmd

โครงสร้างโปรเจกต์ตามหน้าที่

ถ้าจะตาม feature ให้ไล่จาก route/page -> action -> schema -> service -> Prisma model

Key takeawayโค้ดถูกจัดตาม responsibility มากกว่าจัดตามหน้าอย่างเดียวMermaid: scratch/mermaid/project_structure.mmd

Request Lifecycle

ตัวอย่างเส้นทางเมื่อผู้ใช้กดสร้างสินค้าใหม่

Key takeawayทุก mutation ควรผ่าน auth, validation, service transaction, audit และ cache revalidationMermaid: scratch/mermaid/request_lifecycle.mmd

Auth และ RBAC

NextAuth สร้าง session, getSessionContext เติม shop/member/permission ก่อนเข้า service

Key takeawayOwner ข้าม permission ได้ แต่ Member ต้องผ่าน Role.permissionsMermaid: scratch/mermaid/auth_rbac.mmd

Database Domain Map

Shop คือ tenant boundary; domain อื่น ๆ ผูกกับ shopId เพื่อกันข้อมูลข้ามร้าน

Key takeawayอ่าน DB เป็นกลุ่ม domain จะเข้าใจง่ายกว่าอ่านทีละ modelMermaid: scratch/mermaid/db_domain_map.mmd

Multi-Tenant Data Boundary

ทุก service ต้องใช้ ctx.shopId เป็น filter หรือ guard ก่อนอ่าน/เขียนข้อมูลธุรกิจ

Key takeawayshopId คือเส้นแบ่งข้อมูลที่สำคัญที่สุดของระบบMermaid: scratch/mermaid/tenant_boundary.mmd

Inventory: ความจริงของสต็อก

WarehouseStock เป็น source of truth; Product.stock เป็น read cache สำหรับ list/filter/sort

Key takeawayการเปลี่ยน stock ต้องผ่าน StockEngine เพื่อ sync cache และสร้าง StockLogMermaid: scratch/mermaid/inventory_flow.mmd

Sales Flow แบบ End-to-End

หนึ่งบิลขายแตะหลาย subsystem: stock, tax, payment, delivery, accounting และ audit

Key takeawaySaleService ทำ orchestration แต่เรียก core services เป็นเจ้าของกฎเฉพาะทางMermaid: scratch/mermaid/sales_flow.mmd

Sale DB Normalization และ Mapper Shield

Sale ยังมี legacy fields ขณะเดียวกันก็แยก child tables สำหรับ status, tax และ payment

Key takeawayUI ควรอ่านผ่าน SaleMapper เพื่อได้ DTO flat ที่เสถียรMermaid: scratch/mermaid/sale_mapper.mmd

Procurement Flow

จาก PR/PO ไปจนถึง receiving, purchase tax, stock-in และ accounting posting

Key takeawayรับสินค้าเข้าคลังเป็นจุดเชื่อมสำคัญระหว่าง PurchaseReceipt, StockLog และ JournalMermaid: scratch/mermaid/procurement_flow.mmd

Tax + Accounting Flow

เอกสารธุรกิจถูก resolve ภาษี คำนวณยอด และ post เป็น journal entry

Key takeawayComputationEngine คุมเลขเงิน ส่วน PostingService คุม debit/creditMermaid: scratch/mermaid/tax_accounting.mmd

AI และ OCR Flow

API route ใช้ withAuth + rate limit ก่อนเรียก Groq, OCR extraction และ ERP tools

Key takeawayAI ไม่ควรข้าม service layer เพราะ tool execution ยังต้องเคารพ shopId และ permissionMermaid: scratch/mermaid/ai_ocr.mmd

Usage Flow สำหรับการใช้งานจริง

มุมมองผู้ใช้: ตั้งค่าร้าน -> master data -> ซื้อเข้า -> ขายออก -> ปิดบัญชี/รายงาน

Key takeawayflow ธุรกิจหมุนเป็นวงจร ไม่ใช่ feature แยกขาดจากกันMermaid: scratch/mermaid/usage_flow.mmd

อ่านโค้ดหรือ Debug Feature อย่างไร

เส้นทางที่เร็วที่สุดเมื่ออยากเข้าใจหรือแก้บั๊กใน feature ใด feature หนึ่ง

Key takeawayเริ่มจาก route/page แล้วตาม action ไป service และ Prisma modelMermaid: scratch/mermaid/debug_path.mmd

จุดที่ควรระวังในโปรเจกต์ปัจจุบัน

ระบบออกแบบดี แต่มีร่องรอย migration/hardening หลายช่วงที่ควรจำไว้

Docs บางส่วนเก่าเอกสารเดิมพูดถึง 20 models แต่ schema จริงมีมากกว่า
Sale มี legacy + child tablesอ่านผ่าน SaleMapper เพื่อเลี่ยง field ไม่ครบ
Product.stock เป็น cachestock จริงดู WarehouseStock หรือ StockService
Phase/backfill commentsบางโมดูลอยู่ระหว่าง migration และ cleanup
Seed script อาจไม่ตรงpackage.json ชี้ prisma/seed.ts แต่ไฟล์นี้ไม่อยู่ในรายการปัจจุบัน
Test coverage ยังบางควรเพิ่ม service tests สำหรับ sales, stock, accounting
