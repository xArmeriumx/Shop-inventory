3. Solution Overview

Shop Inventory is a web application that runs in any modern browser. It provides a unified dashboard for managing the complete retail workflow from receiving stock from suppliers to shipping products to customers. The system is designed for shop owners and their staff, with different access levels based on assigned roles.

Key Differentiators

  • No installation required: Works entirely in the browser. Accessible from desktop, tablet, and mobile.
  • Multi-channel awareness: Track sales from walk-in, Shopee, Lazada, LINE, Facebook, or custom-defined channels in one place.
  • Financial precision: All monetary calculations use high-precision arithmetic. No rounding errors on invoices or profit reports.
  • Audit trail: Every stock change is logged with timestamp, source, and before/after values. This creates a verifiable history for reconciliation.
  • Data isolation: Each shop's data is completely separated at the database level. No possibility of cross-shop data leakage.
  • Thermal receipt printing: Supports 80mm thermal printers for point-of-sale receipts, complete with items, discounts, and totals.
  • AI-powered receipt scanning: Photograph a supplier invoice and the system extracts items, quantities, and costs automatically.

7. Business Workflows

7.1 Sale Workflow

Select Products ▶ Set Quantities ▶ Apply Discounts ▶ Choose Payment ▶ Confirm Sale ▶ Print Receipt
StepDescription
1. Select ProductsBrowse the product catalog and add items to the cart. Each item shows current stock and sale price.
2. Set QuantitiesAdjust quantity for each item. The system blocks quantities higher than available stock.
3. Apply DiscountsOptional. Two levels available: per-item discount (percentage per line item) and bill-level discount (percentage or fixed amount applied to the total).
4. Choose PaymentSelect Cash (immediately verified), Transfer (requires slip upload and manual verification), or Credit.
5. Confirm SaleSystem generates an invoice number, deducts stock, calculates profit, and saves the transaction.
6. Print ReceiptOptional. Prints an 80mm thermal receipt or allows viewing as a digital receipt.

7.2 Purchase Workflow

Select Supplier ▶ Add Items + Quantities ▶ Set Cost Prices ▶ Confirm Purchase

On confirmation, the system automatically: adds the purchased quantities to stock, updates each product's cost price to the latest purchase cost, and generates a unique purchase number for reference.

7.3 Return Workflow

Open Original Sale ▶ Select Items to Return ▶ Set Return Quantities ▶ Confirm Return

The refund is calculated proportionally — if a bill-level discount was applied to the original sale, the refund per unit accounts for that discount. Stock is restored, and the original sale's net amount and profit are adjusted.

7.4 Shipment Workflow

Pending ▶ Shipped ▶ Delivered

Shipments follow a 5-state lifecycle. Parcels returned by the carrier can be set to "Returned" status and then re-shipped. Pending and Shipped shipments can be cancelled.

9. Financial Controls

9.1 Precision

All monetary calculations use a specialized math library (Decimal.js) that prevents the rounding errors common in standard computer arithmetic. For example, a 15% discount on a 199.50 baht subtotal is calculated precisely without floating-point drift.

9.2 Discount System

LevelTypeDescription
Item LevelPercentageApplied per line item before summing the total. Useful for clearance or loyalty pricing.
Bill LevelPercentage or FixedApplied to the total after item discounts. Used for store-wide promotions or negotiated deals.

9.3 Profit Tracking

Every sale calculates profit automatically: Profit = Net Amount minus Total Cost. The cost is based on the most recent purchase price for each product. This gives the shop owner real-time visibility into margins per transaction and per product.

9.4 Data Integrity

  • Financial records (sales, purchases) are never permanently deleted. They are cancelled with a reason and timestamp, preserving the audit trail.
  • Stock changes are logged individually. Every increment and decrement has a traceable source (sale, purchase, return, adjustment, cancellation).
  • Payment verification for transfers requires a staff member with explicit "Payment Verify" permission to approve or reject the payment proof.

10. Data Protection and Security

MeasureDescription
Data IsolationEach shop's data is stored separately. All database queries are automatically scoped to the logged-in user's shop. No cross-shop access is possible.
AuthenticationUser passwords are hashed and never stored in plain text. Sessions are managed using JSON Web Tokens with expiration.
Input ValidationAll data submitted by users is validated and sanitized before being saved. This prevents injection attacks and data corruption.
Access ControlEvery operation checks the user's role and permissions before executing. Unauthorized access attempts are blocked at the server level.
HTTPSAll communication between the browser and server is encrypted using TLS.
Concurrency SafetyTwo staff members selling the last product simultaneously will not result in negative stock. The system uses atomic operations to prevent overselling.

12. Risk Assessment

RiskLikelihoodImpactMitigation
Internet outage prevents access Medium High The system is cloud-based and requires internet. A future PWA offline mode could cache the product catalog for viewing during outages.
Supabase service disruption Low High Supabase provides 99.9% uptime SLA. Automated backups are enabled. Database can be migrated to another PostgreSQL provider if needed.
Staff misuse of permissions Medium Medium Granular RBAC limits exposure. Audit logs track all actions. Owners should create minimal-privilege roles.
Data growth slows queries Low Low Database indexes are optimized for multi-tenant queries. Soft-deleted records could be archived periodically for very high-volume shops.
Dependency vulnerabilities Medium Medium Regularly run npm audit and update dependencies. GitHub Dependabot can automate vulnerability alerts.

13. Glossary

TermDefinition
POSPoint of Sale. The system module used to create sales transactions.
SKUStock Keeping Unit. A unique identifier for each product in the catalog.
RBACRole-Based Access Control. A security model where permissions are assigned to roles, and roles are assigned to users.
Multi-TenantA system that serves multiple independent shops from a single application, with each shop's data isolated from others.
Invoice NumberA sequential, unique identifier generated for each sale (format: INV-00001, INV-00002, etc.)
Bill-Level DiscountA discount applied to the overall sale total, after individual item discounts have been calculated.
Item-Level DiscountA percentage discount applied to a single line item before the bill total is computed.
Soft DeleteA record is not permanently removed from the database. Instead, a "deleted at" timestamp is set, and the record is hidden from normal views but remains available for audit or recovery.
Payment SlipA photo of the bank transfer receipt uploaded by the customer as proof of payment.
Cost PriceThe price at which the shop purchased a product from a supplier. Used to calculate profit margins.
Net AmountThe final amount after all discounts have been applied. This is the actual revenue recorded for the sale.
OCROptical Character Recognition. Technology that reads text from images (used for receipt scanning).
Thermal PrinterA receipt printer commonly used at retail counters. This system supports 80mm thermal paper format.
