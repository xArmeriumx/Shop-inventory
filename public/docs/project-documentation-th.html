1. ภาพรวมระบบ

Shop Inventory คือระบบ POS และจัดการสต็อกสินค้าผ่านเว็บ ออกแบบสำหรับร้านค้าปลีกขนาดเล็กถึงกลาง ผู้ใช้แต่ละคนเป็นเจ้าของร้านเดียว ข้อมูลแต่ละร้านถูกแยกออกจากกันในระดับฐานข้อมูล ระบบครอบคลุมตั้งแต่แคตตาล็อกสินค้า, การซื้อจากซัพพลายเออร์, การขาย, การจัดส่ง, การคืนสินค้า, รายจ่าย-รายรับ ไปจนถึงรายงานการเงิน

3. สถาปัตยกรรมและ Request Lifecycle

ระบบใช้ Server Actions pattern — ไม่มี REST หรือ GraphQL API แยก ทุก mutation และ query ทำงานผ่าน Next.js Server Actions ที่รันบน server แล้วเรียกตรงจาก React client component

ลำดับการทำงาน

Browser (React)▶ Server Action▶ Auth Guard▶ Zod Validation▶ Business Logic▶ Prisma / DB

ทุก server action เริ่มด้วยการเรียก auth-guard ซึ่งตรวจ session, ค้นหาสมาชิกร้าน, ตรวจสิทธิ์ แล้วคืน SessionContext ที่มี userId, shopId, role, permissions Context นี้ถูกใช้กรอง query ทุกอันให้ตรงกับร้านที่ถูกต้อง

5. ระบบสิทธิ์และการแยกข้อมูล

ขั้นตอนตรวจสิทธิ์

เข้าสู่ระบบ▶ NextAuth JWT Session▶ getSessionContext()▶ SessionContext {userId, shopId, role, permissions}

การทำงานของ Permission

ทุก server action เรียก requirePermission('PERMISSION_NAME') ก่อนทำงานใดๆ ตามลำดับ:

  1. ตรวจว่า session ถูกต้องและยังไม่หมดอายุ
  2. ค้นหา ShopMember record และ Role ที่เชื่อมกัน
  3. ถ้าเป็น OWNER ข้ามการตรวจสิทธิ์ทั้งหมด (เข้าถึงได้ทุกอย่าง)
  4. ถ้าเป็น MEMBER ตรวจว่า role.permissions มีสิทธิ์ที่ต้องการ
  5. คืน SessionContext พร้อม shopId สำหรับกรอง query ทั้งหมด

6. วงจรการขาย

6.1 สร้างรายการขาย

ขั้นตอนการทำงานรายละเอียด
1สร้างเลขใบเสร็จgenerateInvoiceNumber() รูปแบบ INV-00001 มี retry 5 ครั้งกรณี race condition
2ตรวจสอบสินค้าดึงสินค้าทั้งหมดตาม ID ปฏิเสธถ้าไม่มีหรือ isActive = false
3จองสต็อก (atomic)ใช้ updateMany({ where: { id, stock: { gte: qty } }, data: { stock: { decrement: qty } } }) ถ้า count = 0 คือสต็อกไม่พอ ยกเลิก transaction
4คำนวณยอดDecimal.js: subtotal รายการ, ส่วนลดรายการ (G4), ส่วนลดบิล (G4), netAmount, totalCost, profit
5บันทึกลง DBสร้าง Sale + SaleItems ใน Prisma transaction เดียว
6บันทึก AuditStockService.recordMovement() สร้าง StockLog ทุกรายการ + อัปเดต isLowStock
7แจ้งเตือนถ้าสต็อกต่ำกว่า minStock สร้าง LOW_STOCK notification

6.2 การชำระเงิน

วิธีสถานะเริ่มต้นขั้นตอน
เงินสดVERIFIEDยืนยันทันที
โอน/เครดิตPENDINGลูกค้าอัปโหลดสลิป → พนักงานที่มี PAYMENT_VERIFY ตรวจสอบ → ยืนยันหรือปฏิเสธ

6.3 ยกเลิกการขาย

ดำเนินการใน DB transaction เดียว:

  1. เปลี่ยนสถานะ Shipment ทั้งหมดเป็น CANCELLED
  2. คืนสต็อกแต่ละรายการ: คืน quantity - returnedQty (หักส่วนที่คืนไปแล้ว)
  3. ลบรายจ่ายค่าจัดส่งอัตโนมัติ
  4. ตั้ง status = CANCELLED, บันทึก cancelledAt, cancelledBy, cancelReason

7. วงจรการซื้อ

7.1 สร้างรายการซื้อ

ขั้นตอนการทำงานรายละเอียด
1ตรวจ inputZod purchaseSchema ตรวจ items, supplierId, paymentMethod, notes
2สร้างเลขที่รูปแบบ PUR-00001 retry 5 ครั้ง
3คำนวณต้นทุนtotalCost = SUM(quantity * costPrice) ด้วย Decimal.js
4บันทึกPurchase + PurchaseItems ใน transaction เดียว
5เพิ่มสต็อกStockService.recordMovement(type: PURCHASE, qty: +N)
6อัปเดตทุนProduct.costPrice อัปเดตเป็นราคาซื้อล่าสุด

7.2 ยกเลิกการซื้อ

  1. ตรวจสต็อก: สต็อกปัจจุบันต้อง >= จำนวนที่ซื้อ (ยกเลิกไม่ได้ถ้าขายไปแล้ว)
  2. คืนสต็อก: StockService.recordMovement(type: PURCHASE_CANCEL, qty: -N)
  3. คืนราคาทุน: ค้นหาการซื้อก่อนหน้า คืน Product.costPrice เป็นราคาเดิม
  4. เปลี่ยนสถานะ: status = CANCELLED พร้อมเหตุผลและเวลา

8. การจัดการสต็อกและ Audit Trail

ทุกการเปลี่ยนแปลงสต็อกผ่าน StockService.recordMovement() รับประกัน 3 อย่าง: atomic update ในฐานข้อมูล, audit trail ผ่าน StockLog และตรวจสต็อกต่ำอัตโนมัติ

ประเภทการเคลื่อนไหว

ประเภทเกิดจากผลต่อสต็อกจำนวน

PURCHASEสร้างรายการซื้อเพิ่มขึ้น+N

ADJUSTMENTปรับด้วยมือขึ้นหรือลง+/-N

Concurrency Control

การจองสต็อกตอนขายใช้ updateMany เดียวพร้อม WHERE stock >= qty เป็น atomic compare-and-decrement ถ้ามีคนอื่นตัดสต็อกก่อน result.count จะได้ 0 วิธีนี้ปลอดภัยกว่า SELECT แยกแล้วค่อย UPDATE เพราะไม่มีช่องว่างให้ race condition เกิด

9. การคำนวณเงิน

9.1 คำนวณยอดขาย

// ทั้งหมดใช้ Decimal.js
SaleItem.subtotal    = quantity * salePrice
SaleItem.subtotal   -= itemDiscountAmount            // G4 ส่วนลดรายการ

Sale.totalAmount     = SUM(ทุก subtotal)
Sale.discountAmount  = totalAmount * (percent / 100)  // หรือจำนวนคงที่
Sale.netAmount       = totalAmount - discountAmount   // G4 ส่วนลดบิล

Sale.totalCost       = SUM(quantity * costPrice)
Sale.profit          = netAmount - totalCost

9.2 คำนวณเงินคืน

// เงินคืนตามสัดส่วนส่วนลดบิล
billDiscountRatio    = Sale.discountAmount / Sale.totalAmount
refundPerUnit        = salePrice * (1 - billDiscountRatio)
ReturnItem.refund    = returnQty * refundPerUnit

// ปรับยอด Sale ต้นทาง
Sale.netAmount      -= totalRefund
Sale.totalCost      -= (returnQty * costPrice)
Sale.profit          = netAmount - totalCost          // คำนวณใหม่

12. กลยุทธ์การลบข้อมูล

รูปแบบพฤติกรรมใช้กับ
Cascadeลบแม่ ลูกหายตามอัตโนมัติShop (ข้อมูลทั้งร้าน), Sale→Items, Purchase→Items, Return→Items
Restrictห้ามลบแม่ถ้ายังมีลูกUser→Shop (ห้ามลบเจ้าของร้าน)
SetNullFK ตั้งเป็น NULL เมื่อข้อมูลอ้างอิงถูกลบStockLog.saleId, purchaseId, returnId
Soft Deleteใส่ deletedAt ซ่อนจาก query (ไม่ลบจริง)Product, Customer, Supplier, Expense, Income

Sale และ Purchase ไม่ถูกลบจริง ใช้ status = CANCELLED พร้อม audit trail (cancelledAt, cancelledBy, cancelReason) เพื่อรักษาความถูกต้องของข้อมูลการเงิน

สำคัญ: ความสัมพันธ์ User-Shop

ใช้ onDelete: Restrict ถ้าพยายามลบ User ที่เป็นเจ้าของร้าน Prisma จะปฏิเสธในระดับ database constraint ต้องลบหรือโอนร้านก่อน แก้ไขจาก Cascade เดิม (ซึ่งจะทำลายข้อมูลทั้งร้าน)

13. ความปลอดภัย

13.1 การป้องกัน XSS

ทุก text field ที่ผู้ใช้กรอกผ่าน sanitizeText() ใน Zod .transform() ก่อนถึงฐานข้อมูล ครอบคลุม: ชื่อสินค้า, ชื่อลูกค้า, ที่อยู่, หมายเหตุ, คำอธิบาย ใน schema ทั้ง 9 ไฟล์

13.2 การป้องกัน Race Condition

กลไกที่ใช้ทำไม
Atomic updateMany + guard clauseจองสต็อกตอนขายป้องกันขายเกินด้วย check ใน UPDATE เดียว
Optimistic locking (version)แก้ไขสินค้าตรวจจับการแก้ไขพร้อมกัน version ไม่ตรงก็ fail
Retry 5 ครั้ง + offsetเลขใบเสร็จ/ซื้อ/ส่ง/คืนกรณีสร้างเลขพร้อมกันแล้วชนกัน
Prisma $transactionทุกการทำงานหลายขั้นตอนสำเร็จทั้งหมดหรือไม่ทำเลย

15. ปัญหาที่พบและแนวทางดูแลระบบ

แก้ไขแล้ว (กุมภาพันธ์ 2569)

ระดับปัญหาวิธีแก้
วิกฤตUser-Shop ใช้ onDelete: Cascade ลบ user ข้อมูลทั้งร้านหายเปลี่ยนเป็น onDelete: Restrict
ปานกลางPurchase paymentMethod ผ่าน Zod แต่ไม่มีใน Prisma model ข้อมูลหายตอน saveเพิ่ม field ใน Prisma model พร้อม @default("CASH")
ปานกลางShipment/CustomerAddress schema ขาด sanitizeText() เสี่ยง XSSเพิ่ม .transform(sanitizeText) ทุก text field

รอแก้ไข

ระดับปัญหาคำแนะนำ
ปานกลางverifyPayment ใช้ .uuid() ตรวจ ID แต่ระบบใช้ cuid()เปลี่ยนเป็น .cuid() หรือ .min(1)
ต่ำStockLog ใน createSale ใช้ referenceId แทน FK saleIdใช้ saleId ตรงๆ

แนวทางดูแลระบบ (Maintenance)

  • เพิ่ม field ใหม่: แก้ Prisma schema → npx prisma migrate dev → แก้ Zod schema → แก้ server action
  • เพิ่ม permission ใหม่: เพิ่มใน lib/constants.ts → ใช้ requirePermission() ใน action → อัปเดต UI จัดการ role
  • แก้ไขสต็อก: ห้ามแก้ product.stock ด้วย Prisma ตรงๆ ต้องผ่าน StockService.recordMovement() เสมอ ไม่งั้น audit log และ low-stock จะไม่ทำงาน
  • คำนวณเงิน: ใช้ Decimal.js เท่านั้น ห้ามใช้ JavaScript arithmetic กับค่าเงิน แปลงเป็น number ด้วย toNumber() ตอน output สุดท้าย
  • Migration production: ใช้ npx prisma migrate deploy (ไม่ใช่ dev) ทดสอบ migration กับ staging ก่อนเสมอ
