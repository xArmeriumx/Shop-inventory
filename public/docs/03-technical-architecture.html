1. System Overview

Shop Inventory is a web-based POS and inventory management platform designed for small-to-medium retail businesses. Each user account owns a single shop, and shop data is fully isolated at the database level. The system handles the entire retail workflow: product catalog, purchasing from suppliers, point-of-sale transactions, shipment tracking, returns, expense/income logging, and financial reporting.

Core Capabilities

  • Point of Sale with item-level and bill-level discounts (referred to as G4 pattern internally)
  • Multi-channel sales tracking: walk-in, Shopee, Lazada, LINE, Facebook, and custom channels
  • Payment verification workflow with slip image upload for transfers
  • Partial returns with proportional refund calculation (G3 pattern)
  • Shipment lifecycle tracking with a 5-state finite state machine
  • Purchase management with automatic cost price synchronization
  • Real-time stock tracking with low-stock alerts and audit logging
  • Expense and income logging with dashboard profit/loss calculations
  • Customer and supplier relationship management
  • Thermal receipt printing (80mm format)
  • Receipt OCR via Tesseract.js and Groq AI for purchase data extraction

3. Architecture & Request Lifecycle

The application follows a Server Actions pattern — there is no separate REST or GraphQL API layer. All mutations and queries are handled through Next.js Server Actions that run on the server and are invoked directly from React client components.

Request Pipeline

Browser (React) ▶ Server Action ▶ Auth Guard ▶ Zod Validation ▶ Business Logic ▶ Prisma / DB

Every server action begins with an auth-guard call that verifies the session, resolves the user's shop membership, checks permissions, and returns a SessionContext object containing userId, shopId, role, and permissions. This context is then used to scope all database queries to the correct tenant.

5. RBAC & Tenant Isolation

Authentication Flow

Credentials Login ▶ NextAuth JWT Session ▶ getSessionContext() ▶ SessionContext {userId, shopId, role, permissions}

Permission Resolution

Every server action calls requirePermission('PERMISSION_NAME') before executing any logic. This function performs the following checks in order:

  1. Verifies the user has a valid, non-expired session.
  2. Resolves the user's ShopMember record and associated Role.
  3. If the user is the shop OWNER, all permission checks are bypassed — owners have unrestricted access.
  4. If the user is a MEMBER, the function checks whether role.permissions contains the required permission string.
  5. On success, returns a SessionContext with the resolved shopId, which is used to scope all subsequent database queries.

Tenant Isolation

Every database query includes a WHERE shopId = ctx.shopId clause. The shopId is never accepted from client input — it is always resolved server-side from the authenticated session. This guarantees that no user can access another shop's data regardless of the request payload.

6. Sale Lifecycle

6.1 Creating a Sale

StepOperationImplementation Detail
1Generate invoice numbergenerateInvoiceNumber() produces sequential numbers in the format INV-00001. Uses retry logic (up to 5 attempts) with incremental offset to handle race conditions when multiple sales are created simultaneously.
2Validate productsFetches all referenced products by ID. Rejects the request if any product does not exist or has isActive = false.
3Reserve stock atomicallyUses prisma.product.updateMany({ where: { id, stock: { gte: qty } }, data: { stock: { decrement: qty } } }). If the affected row count is 0, the stock was insufficient — the transaction is aborted.
4Calculate financialsAll arithmetic uses Decimal.js. Computes item subtotals, applies G4 item-level discounts, sums to totalAmount, applies bill-level discount, derives netAmount, totalCost, and profit.
5Persist to databaseCreates the Sale record with nested SaleItem creation inside a single Prisma transaction.
6Record stock movementsCalls StockService.recordMovement() for each item, which creates a StockLog entry and updates the product's isLowStock flag.
7Trigger notificationsIf any product's stock drops below minStock, a LOW_STOCK notification is created.

6.3 Cancelling a Sale

Cancellation is a multi-step operation executed within a database transaction:

  1. All associated Shipment records are set to status = CANCELLED.
  2. Stock is restored for each item: the quantity restored is quantity - returnedQty, accounting for any partial returns that have already been processed.
  3. Any auto-generated shipping expense records linked to the sale are deleted.
  4. The sale record is updated: status = CANCELLED, cancelledAt, cancelledBy, and cancelReason are set.

7. Purchase Lifecycle

7.1 Creating a Purchase

StepOperationDetail
1Validate inputZod purchaseSchema validates item array, supplierId, paymentMethod, and notes.
2Generate purchase numberFormat: PUR-00001. Same retry pattern as invoice numbers.
3Calculate total costtotalCost = sum(quantity * costPrice) using Decimal.js.
4Persist recordsPurchase + PurchaseItems created in a single Prisma transaction.
5Increment stockStockService.recordMovement(type: PURCHASE, qty: +N) for each item.
6Sync cost priceEach product's costPrice is updated to the latest purchase cost.

7.2 Cancelling a Purchase

  1. Stock validation: Verifies that current stock is greater than or equal to the originally purchased quantity. If stock has already been sold, cancellation is blocked.
  2. Stock reversal: StockService.recordMovement(type: PURCHASE_CANCEL, qty: -N) for each item.
  3. Cost price reversion: Finds the most recent previous purchase of the same product and reverts Product.costPrice to that value. If no prior purchase exists, the cost price remains unchanged.
  4. Status update: status = CANCELLED with reason and timestamp.

8. Stock Management & Audit Trail

All stock modifications pass through StockService.recordMovement(). This centralization provides three guarantees: atomic database updates, a complete audit trail via StockLog, and automatic low-stock detection.

Movement Types

TypeTriggerEffectQuantity
SALESale creationStock decreases-N
PURCHASEPurchase creationStock increases+N
RETURNReturn creationStock increases+N
ADJUSTMENTManual adjustmentStock increases or decreases+/-N
SALE_CANCELSale cancellationStock increases+N

Concurrency Control

Stock reservation during sale creation uses a single updateMany query with a WHERE stock >= requestedQty guard. This is an atomic compare-and-decrement operation — if another transaction reduces the stock first, result.count returns 0. This pattern is more reliable than a separate SELECT followed by UPDATE, as it eliminates the TOCTOU (time-of-check-to-time-of-use) race window entirely.

9. Financial Calculations

9.1 Sale Total Computation

// All values computed with Decimal.js to avoid floating-point errors.
SaleItem.subtotal    = quantity * salePrice
SaleItem.subtotal   -= itemDiscountAmount            // G4 item-level discount

Sale.totalAmount     = SUM(all item subtotals)
Sale.discountAmount  = totalAmount * (percent / 100)  // or fixed value
Sale.netAmount       = totalAmount - discountAmount   // G4 bill-level discount

Sale.totalCost       = SUM(quantity * costPrice)
Sale.profit          = netAmount - totalCost

9.2 Return Refund Computation

// Proportional refund that accounts for bill-level discount already applied.
billDiscountRatio    = Sale.discountAmount / Sale.totalAmount
refundPerUnit        = SaleItem.salePrice * (1 - billDiscountRatio)
ReturnItem.refund    = returnQty * refundPerUnit

// Original sale record is adjusted:
Sale.netAmount      -= totalRefund
Sale.totalCost      -= (returnQty * costPrice)
Sale.profit          = netAmount - totalCost          // recalculated

12. Data Deletion Strategy

The system uses four distinct patterns for data removal, chosen based on the data's importance and relationships:

PatternBehaviorApplied To
CascadeDeleting the parent automatically deletes all children. Used for data that has no meaning without its parent.Shop (all data), Sale items, Purchase items, Return items
RestrictDeletion of the parent is blocked if any children exist. Prevents accidental data loss.User to Shop — the shop owner cannot be deleted while the shop exists.
SetNullThe foreign key is set to NULL when the referenced record is removed. Preserves the child record for audit purposes.StockLog references (saleId, purchaseId, returnId)
Soft DeleteThe record's deletedAt field is set to the current timestamp. It remains in the database but is excluded from all queries.Product, Customer, Supplier, Expense, Income, CustomerAddress

Financial transaction records (Sales, Purchases) are never deleted. They are voided by setting status = CANCELLED with an audit trail (cancelledAt, cancelledBy, cancelReason). This ensures financial data integrity for reporting and reconciliation.

Critical: User-Shop Relationship

The User to Shop relation uses onDelete: Restrict. If someone attempts to delete a user who owns a shop, Prisma will reject the operation at the database constraint level. The owner must explicitly delete or transfer the shop first. This was changed from Cascade (which would have destroyed all shop data) as a critical safety fix.

13. Security Model

13.1 Input Sanitization

All user-facing text fields pass through sanitizeText() via Zod's .transform() pipeline before reaching the database. This includes product names, customer names, addresses, notes, and descriptions across all 9 schema files. The sanitizer encodes HTML special characters to prevent stored XSS attacks.

13.2 Concurrency Safeguards

MechanismLocationPurpose
Atomic updateMany with guard clauseStock reservation in createSalePrevents overselling by checking stock level within the same UPDATE statement.
Optimistic locking (version field)Product updatesDetects concurrent modifications. If the version does not match, the update fails and the user must retry.
Sequential number retry (5 attempts)Invoice, purchase, shipment, return numbersHandles race conditions where two transactions attempt to claim the same sequential number.
Prisma $transactionAll multi-record operationsEnsures atomicity — either all records are written or none are.

Maintenance Guidelines

  • Adding a new field: Update the Prisma schema, create a migration (npx prisma migrate dev), update the Zod validation schema, and update the corresponding server action.
  • Adding a new permission: Add the permission string to lib/constants.ts, use it in the relevant server action via requirePermission(), and update the role management UI.
  • Modifying stock logic: Always go through StockService.recordMovement(). Direct Prisma updates to the product stock field will bypass audit logging and low-stock detection.
  • Financial calculation changes: Use Decimal.js for all arithmetic. Never use native JavaScript arithmetic on monetary values. Convert to number only at the final output boundary using toNumber().
  • Running migrations in production: Use npx prisma migrate deploy (not dev). Always test migrations against a staging database first.
